# TRACE — 会话后自动复盘系统

TRACE (Transactional Reflection and Analysis for Collaborative Effectiveness) 是一个 Claude Code 技能，在每次开发 session 结束时自动进行结构化复盘，分析协作质量，并将经验教训持久化到跨会话记忆，实现持续自我改进。

## 痛点

在长时间的开发项目中（通常超过 10k 上下文），仅靠人类记忆很难系统性地总结整个开发流程中的缺陷和经验。常见的困境：

- 一个项目开发完成后，踩过的坑很快就忘了
- 同样的问题在多个 session 中反复出现
- 用户的提问方式有没有可以改进的地方？Agent 的思考有没有盲点？没有人系统性记录
- 经验教训只能靠自觉总结，没有结构化的收集机制

## 核心思想

TRACE 的核心理念借鉴自 Hermes Agent 的自我改进机制：**每次协作都是一次学习机会，关键是把它提取出来。**

与 Claude Code 内置记忆系统的区别：

| | 内置记忆系统 | TRACE |
|---|---|---|
| **粒度** | 原子事实（"用户偏好 snake_case"） | 过程级分析（"这个 session 因需求模糊导致 40% 返工"） |
| **分析方向** | 单向记录用户偏好 | 双向分析（用户提问方式 + Agent 思考/输出质量） |
| **时机** | 对话中即时记录 | session 结束后整体复盘 |
| **洞察类型** | "记住这个事实" | "这个模式可以改进" |
| **依赖** | 内置功能，零配置 | 一个 SKILL.md 文件 |

## 架构

```
~/.claude/skills/post-session-reflection/
  SKILL.md                  ← 唯一的技能文件（手动创建）

<!-- 以下文件由技能自动维护 -->
C:\Users\<用户名>\CLAUDE.md
  └── 会话后复盘 部分         ← 累积经验教训，每次 session 自动加载

<memory-dir>/reflection-YYYY-MM-DD.md   ← 每次 session 的详细复盘
<memory-dir>/reflection-index.md        ← 所有复盘的索引表格
```

整个系统只有一个手动创建的入口文件（SKILL.md），零外部依赖，不需要 Node.js、MCP 服务器、或任何 API 配置。

## 安装

### 前提条件

- Claude Code 已安装并配置
- 文件系统可写（用于创建技能文件和复盘文件）

### 步骤

1. **创建技能目录：**

```bash
mkdir -p ~/.claude/skills/post-session-reflection
```

2. **下载或创建 SKILL.md：**

将 `SKILL.md` 放置到 `~/.claude/skills/post-session-reflection/` 目录下。

3. **重启 Claude Code：**

技能文件在 session 启动时自动发现。重启后，输入 `/post` 即可看到自动补全。

### 验证安装

重启 Claude Code 后，输入以下命令验证：

```
/post-session-reflection
```

如果技能已正确安装，Claude 会开始执行复盘流程。

## 使用方法

### 基础用法

在每次开发 session 结束时运行：

```
/post-session-reflection
```

Claude 会自动执行完整的复盘流程。

### 复盘流程

技能执行时，Claude 会依次完成以下 5 步：

**第一步：回顾当前 session**

Claude 基于当前上下文中的对话记忆，回顾整段开发过程。聚焦对话中的关键时刻——好的或坏的。

**第二步：分析四个维度**

逐一分析以下维度，每个维度都有具体的分析点和输出：

| 维度 | 分析点 | 输出 |
|------|--------|------|
| 用户提示质量 | 清晰度、完整性、上下文是否充分、反馈是否及时 | 评级 + 1-2 条改进建议 |
| Agent 思考过程 | 是否过早下结论、有无未验证假设、是否遗漏上下文 | 标记好坏模式各 1-2 个 |
| Agent 输出质量 | 准确性、完整性、结构、可验证性 | 评级 + 具体改进点 |
| 流程模式 | 目标匹配度、重复工作、成功模式、可自动化点 | 最多 3 条关键经验教训 |

**第三步：写入复盘文件**

创建 `reflection-YYYY-MM-DD.md` 到 Claude Code 的记忆目录。这个文件包含完整的四维度分析和具体例子，是可供日后查阅的详细记录。

**第四步：更新 CLAUDE.md**

将本次的关键经验教训合并到 `CLAUDE.md` 中的 `## 会话后复盘` 部分。这是跨 session 复用的桥梁——CLAUDE.md 在每个 session 启动时自动加载，所以这些经验教训会在下次开发时自动生效。

合并规则：
- 相似教训合并（如"需求模糊导致返工（第 3 次出现）"）
- 最多保留 10 条，超限时合并去重
- 不删除 CLAUDE.md 中非复盘部分的内容

**第五步：更新复盘索引**

在 `reflection-index.md` 中追加一行摘要，形成所有复盘的结构化历史：

```
| 日期 | session | 用户评级 | AI 输出评级 | 关键经验教训 |
```

## 四个维度的详细说明

### 维度 1：用户提示质量

分析用户（人类搭档）提问的方式：

- **清晰度：** 是否明确表达了想要什么？还是需要多轮追问才能理解？
- **完整性：** 需求是否一次性完整提出？还是边做边改？
- **上下文：** 是否主动提供了文件路径、错误信息、预期结果等必要上下文？
- **反馈：** 是否在结果偏离期望时及时且明确地指出？

**目标：** 帮助用户优化与 AI 协作时的沟通效率。

### 维度 2：Agent 思考过程

分析 Claude 的推理模式：

- **过早结论：** 是否未经充分探索就锁定了方案？
- **未验证假设：** 是否有假设未被核实？
- **上下文遗漏：** 是否忽略了用户明确提供的信息？
- **探索平衡：** 是否在探索多种方案和深入实施之间取得了平衡？

**目标：** 识别思考过程中的系统性盲点。

### 维度 3：Agent 输出质量

分析 Claude 的输出：

- **完整性：** 是否完全解决了用户的问题？
- **准确性：** 代码/方案是否经过了验证？
- **结构：** 输出组织是否清晰、逻辑是否连贯？
- **验证：** 是否主动验证了结果？

**目标：** 量化并提升输出质量。

### 维度 4：流程模式

分析整体协作中的过程性模式：

- **目标匹配：** 双方是否在共同理解的目标上工作？
- **重复劳动：** 是否有来回确认浪费的精力？
- **成功模式：** 本 session 中什么做法特别高效？
- **可自动化：** 有没有可以通过模板/checklist 解决的重复性问题？

**目标：** 发现流程层面的优化机会。

## 跨会话经验复用的工作机制

TRACE 的跨会话复用不需要任何额外配置，完全依托 Claude Code 的固有机制：

```
Session N 结束
  → 运行 /post-session-reflection
    → 关键经验教训写入 CLAUDE.md
      → Session N+1 启动，CLAUDE.md 自动加载
        → Agent 一开始就知道前几个 session 踩过哪些坑
```

**自动生效，无需配置。** Claude Code 在每个 session 启动时自动读取当前目录及用户目录下的 CLAUDE.md。

## 文件说明

### 技能文件

| 路径 | 说明 |
|------|------|
| `~/.claude/skills/post-session-reflection/SKILL.md` | 技能定义。唯一需要手动创建的文件 |

### 复盘产出文件（自动维护）

| 路径 | 说明 |
|------|------|
| `<memory-dir>/reflection-YYYY-MM-DD.md` | 每次 session 的完整复盘记录 |
| `<memory-dir>/reflection-index.md` | 所有复盘的索引，便于追溯 |
| `~/CLAUDE.md` | 累积经验教训，跨 session 复用的桥梁 |

其中 `<memory-dir>` 为 Claude Code 的项目记忆目录，路径格式为：`~/.claude/projects/<项目路径哈希>/memory/`

## 最佳实践

### ✅ 推荐做法

- **每次 session 结束时运行复盘**，形成习惯
- **聚焦关键时刻**，而非试图回忆每个细节
- **记录正向反馈**——什么做得好也值得记录和保持
- **新 session 开始时快速浏览 CLAUDE.md**，知悉之前的经验教训

### ❌ 避免做法

- **不要在 session 中间运行**——数据不足以发现模式
- **不要输出泛泛而谈的分析**——"提示要更清晰" 不如 "提需求时附带具体文件路径"
- **不要重复记录相同教训**——先检查 CLAUDE.md 中已有的教训，合并而非重复
- **不要追求面面俱到**——3-5 个最关键的观察胜过 10 条不痛不痒的记录

## 常见问题

### 技能文件创建后为什么不生效？

技能在 session 启动时自动发现。创建文件后需要**重启 Claude Code**。

### 复盘依赖读 JSONL 文件吗？

不依赖。复盘基于 Claude 当前上下文中的对话记忆，不需要读文件。

### 纯文本模型能使用吗？

能。TRACE 的核心分析基于纯文本对话，不需要多模态能力。如果你使用纯文本模型，对话中本身就不会出现图片。

### 多模态模型能使用吗？

也能。多模态模型可以同时处理文本和图片信息，分析维度不变。

### 复盘内容会占用上下文吗？

复盘写入记忆文件和 CLAUDE.md，不在当前 session 上下文中保留。下次 session 启动时，CLAUDE.md 中的经验教训摘要（约 10 条短句）会加载到上下文中，开销极小。

### 经验教训超过 10 条怎么办？

技能会自动合并相似教训并去重。如果某条教训反复出现，会被标记为"第 N 次出现"并提升优先级。

### 可以自定义分析维度吗？

可以编辑 SKILL.md 调整分析维度。SKILL.md 是标准的 Markdown 文件，按照 Claude Code 技能格式编写即可。

## 与类似方案的区别

| 方案 | 依赖 | 自动化程度 | 跨会话复用 |
|------|------|-----------|-----------|
| **TRACE**（本方案） | 零外部依赖 | 手动触发，自动写文件 | CLAUDE.md 自动加载 |
| Hermes Agent | Python + SQLite + API | 全自动后台审查 | 独立记忆系统 |
| Factory Signals | 闭源商业产品 | 全自动 | 工单 + PR 自动化 |
| 内置记忆系统 | 内置于 Claude Code | 对话中即时记录 | 事实级，非过程级 |

## 技术原理

TRACE 利用 Claude Code 的以下系统能力：

1. **Skill 系统：** `~/.claude/skills/*/SKILL.md` 在 session 启动时被自动发现并注入技能列表
2. **CLAUDE.md 自动加载：** Claude Code 在 session 启动时自动读取项目目录和用户目录下的 CLAUDE.md
3. **记忆系统：** 复盘文件通过 `Write` tool 写入项目记忆目录，长期保存
4. **对话记忆：** Claude 基于当前上下文中的整段对话进行回顾分析，不依赖外部数据源

整个系统的数据流：

```
对话记忆 (in-context)
  → 四维度分析 (Claude 推理)
    → Write tool → 复盘文件 (memory/)
    → Write/Edit tool → CLAUDE.md (跨session)
    → Write/Edit tool → 复盘索引 (memory/)
```
